### Active-user ownership

- `DashboardView` holds `@AppStorage(AppStorageKey.activeUserId)` — sole owner of active-user selection
- VM receives `activeUserId` on `loadToday(context:activeUserId:)`; never reads/writes `UserDefaults` for profile switching
- Profile switcher sets `@AppStorage` in the view; `.onChange(of: activeUserId)` reloads the VM
- After load, if `activeUserId` is empty, view backfills from `activeProfile.id` (guards onboarding/`@AppStorage` drift without double reload)

### Data loading

```
loadToday(context:activeUserId:)
  → fetchAll profiles
  → resolve activeProfile (UUID match or profiles.first)
  → fetch today's meals → aggregate totals
  → fetch chart weigh-ins (last 7 calendar days)
  → fetch latest 3 weigh-ins (plateau proxy)
  → checkForPlateau()
```

### Plateau actions (field sync rule)

Both mutations update **`dailyCalorieTarget`**, **`deficitKcal`**, and **`updatedAt`** together, then `context.save()`:

| Action | Effect |
|--------|--------|
| **Diet Break** | `dailyCalorieTarget = tdee`, `deficitKcal = 0`; store `maintenanceModeUntil_{uuid}` +14 days |
| **Small Reduction** | `dailyCalorieTarget = max(minimum, current - 60)`, `deficitKcal = tdee - dailyCalorieTarget` |
| **Dismiss** | store `plateauSnoozeUntil_{uuid}` +14 days |

Plateau AppStorage keys always use **`activeProfile.id`**, not the view’s `activeUserId` string.

`checkForPlateau()` suppresses alert while maintenance or snooze is active.

## 9. Risks and edge cases

| Risk | Mitigation |
|------|------------|
| Stale dashboard after profile switch | `.onChange(of: activeUserId)` reloads VM |
| Stale meals on failed fetch | `clearTodayData()` at start of each `loadToday`; `showPlateauAlert = false` on catch |
| Empty `@AppStorage` after onboarding | VM fallback to `profiles.first`; view backfills `activeUserId` after load |
| Plateau keys stored under wrong UUID | Keys use `activeProfile.id` in all plateau actions |
| Plateau proxy vs PR6 weekly cadence | PR3 uses last 3 recorded weigh-ins (any spacing); PR6 replaces with weekly cadence |
| Maintenance mode never auto-expires | Documented limitation; PR6/Settings can restore deficit |
| Weight chart with 0–1 points in 7-day window | Empty-state card with starting weight label |
| Swipe-dismiss plateau sheet re-shows | `.interactiveDismissDisabled()` on sheet |
| `xcodegen generate` wipes project | Edit `CalSnap.xcodeproj` directly until `project.yml` updated |

### Spec extensions (document in PR description)

1. **`loadToday(context:activeUserId:)`** — view owns `@AppStorage`; VM does not read/write active user from `UserDefaults`.
2. **`@MainActor` on `DashboardViewModel`** — matches `OnboardingViewModel`; spec snippet omitted it.
3. **Weight chart** — weigh-ins from last **7 calendar days**, not latest 7 records by count.
4. **Plateau detection** — PR3 proxy using last 3 recorded weigh-ins; PR6 adds true weekly cadence.
5. **Plateau actions** — update `dailyCalorieTarget`, `deficitKcal`, `updatedAt` atomically; maintenance suppresses re-alert.
6. **Weight display** — locale default via `Locale.current.measurementSystem`; onboarding imperial toggles not persisted until Settings.
7. **Meal rows** — display-only; `MealDetailView` deferred to PR5.
8. **Plateau AppStorage** — snooze/maintenance keyed by user UUID; no model schema change.
9. **Project maintenance** — PR3 file references edited in `CalSnap.xcodeproj`; `project.yml` not updated.
