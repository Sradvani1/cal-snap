## 1. Objective

Deliver the first-run onboarding experience for a **single local profile**: collect body metrics and goals, calculate calorie and macro targets via `NutritionCalculator`, request HealthKit authorization, optionally validate and store API keys in Keychain, and persist one `UserProfile` to SwiftData. After completion, `RootView` gates to a minimal `DashboardView` stub; relaunch skips onboarding when a profile exists.

**Single-user addendum:** No partner loop, no required display name on Welcome/Profile Setup. Neutral step titles (`"Your profile"`, `"Your goals"`, `"Your calorie target"`). See [`PR-single-user-local-only-addendum.md`](./PR-single-user-local-only-addendum.md).

## 2. In scope

- **Launch gating** — `RootView` uses `@Query` on `UserProfile`; empty → `OnboardingView`, else → `DashboardView`
- **Seven-step onboarding flow** — Welcome, Profile Setup, Goal Setup, Calorie Target Preview, HealthKit, API Keys, Done
- **`OnboardingViewModel`** — Linear step navigation; single `profileDraft`; validation, target calculation, HealthKit request, API key save, profile persistence
- **`ProfileDraft`** — In-memory draft struct; converted to `UserProfile` via repository (display name optional)
- **`UserProfileRepository`** — `makeUserProfile(from:)`, `save(_:context:)`, `fetchAll`, **`fetchPrimaryProfile(context:)`** (earliest `createdAt`, `fetchLimit = 1`)
- **`AppContainer` wiring** — Concrete `HealthKitService`, `GeminiService`, `UserProfileRepository`; injected via `.environment(appContainer)`
- **`HealthKitService`** — Authorization only (read body mass/height; write dietary + body mass)
- **`GeminiService`** — `validateAPIKey(_:)` only (trivial `generateContent` call); **Gemini key skippable** on Continue
- **`APIKeyResolver`** — Keychain-first Gemini/USDA resolution; USDA falls back to `AppConstants.USDA.demoAPIKey`
- **`UnitFormatters`** — kg/lbs and cm/ft-in conversion for onboarding pickers
- **`UserProfile+Display`** — Display helpers for imperial units
- **`AppConstants.Onboarding`** — Age 18–90; goal date 14–730 days from today
- **HealthKit entitlements + Info.plist usage strings**
- **SPM:** `generative-ai-swift` (`GoogleGenerativeAI`) with `Package.resolved` committed
- **Unit tests:** onboarding validation (name not required), goal date minimum, profile persistence
- **Post-review fixes:** atomic profile save; README xcodegen warning; `OnboardingViewModelTests` unwraps profile before `accuracy` assertions
- **Skills alignment:** per-profile unit toggles on `ProfileDraft`; `NavigationStack` wrapper; extracted `OnboardingStepContent`, `OnboardingNavigationBar`, `GeminiTestIndicatorView`, `GeminiTestState`; VM binding helpers; profile-before-keys save order; HealthKit error surfacing; `AppConstants.Nutrition` macro defaults

11. Add `OnboardingViewModelTests` (three cases)

13. Build & test: `DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer xcodebuild -scheme CalSnap -destination 'platform=iOS Simulator,name=iPhone 16' test`

### Onboarding navigation (single user)

```
welcome → profileSetup → goalSetup → caloriePreview → healthKit → apiKeys → [save] → done
```

### Persistence timing (step 6 Continue)

On API Keys Continue:

1. `saveProfile(context:)` — insert one profile, `context.save()`
2. `saveAPIKeys()` — Keychain only for non-empty trimmed inputs (Gemini optional)
3. Advance to `.done` (RootView immediately shows dashboard via `@Query`)

## 9. Risks and edge cases

| Risk | Mitigation |
|------|------------|
| Legacy multi-profile stores | `fetchPrimaryProfile` picks earliest `createdAt`; no switcher UI |
| `xcodegen generate` wipes PR2 project settings | Documented in README; edit `CalSnap.xcodeproj` directly until `project.yml` updated |
| Gemini SDK spec timing (PR4 in original spec) | Approved pull-forward: validate-only in PR2; `analyzeMeal` deferred |
| Step 7 Done rarely visible | `@Query` in `RootView` updates as soon as profiles persist; Done is transient |
| Keys saved but profiles fail | Profiles save first; Keychain write happens only after successful SwiftData save |
| Profiles saved but keys fail | Onboarding completes; `validationError` shows keychain message if user entered keys (RootView may already show dashboard) |
| Xcode license / CI | Local verify with `DEVELOPER_DIR=… xcodebuild test` after accepting license |

### Spec extensions (document in PR description)

1. **`generative-ai-swift` in PR2** — SDK added early for `validateAPIKey` only; meal analysis remains PR4+.
2. **Gemini key skippable** — User can finish onboarding without entering or testing a Gemini key.
3. **`USDA.demoAPIKey`** — `"DEMO_KEY"` constant; Keychain used only when user enters a key; `APIKeyResolver` provides runtime fallback.
4. **`profileDraft`** — Single in-memory draft; no `profileA` / `profileB` loop.
5. **`UserProfileRepository.save`** — Persists one profile per onboarding completion.
6. **Project maintenance** — PR2 SPM, entitlements, and file references edited in `CalSnap.xcodeproj`; `project.yml` not updated in this PR.
