## 1. Objective

Replace the PR3 `MealScannerView` stub with the full meal-logging flow: camera/photo capture, optional description, Gemini structured meal analysis, editable results with proportional macro rescaling, manual entry fallback, SwiftData persistence, fire-and-forget HealthKit dietary writes, and **storage-optimized local persistence of captured/imported meal photos**. Business logic lives in `MealScannerViewModel`; views stay thin.

## 6. Key implementation details

### Phase flow

```
capture → (Analyze) → analyzing → results
capture/error → manual → results
error states: offline, missing API key, API, parse, unrecognizable
```

### Manual confidence semantics (approved)

| Field | Manual entry value |
|-------|-------------------|
| Per-item `confidence` | `1.0` |
| Per-item `isFlagged` | `false` |
| `MealEntry.geminiConfidence` | `0` |
| `MealEntry.isManuallyAdjusted` | `true` |
| `estimationNotes` | `nil` |
| Results UI | "Manual entry" label (not confidence badge) |

### Totals

Displayed and persisted totals are **sums of `editableItems`**, not Gemini `meal_total` (after user edits).

### Camera unavailable

`MealScannerViewModel.isCameraAvailable` hides Take Photo; inline caption; library + manual entry remain available (simulator QA path).

### HealthKit

`logMeal` fires `Task { await healthKitService.logMeal(snapshot) }` after SwiftData save — never blocks UI; errors logged to console only. Uses `Sendable` `MealHealthSnapshot` across unstructured `Task`.

### Active user

`DashboardView` passes `activeUserId` to `MealScannerView` via `DashboardContentView`. Invalid/empty UUID shows `ContentUnavailableView`.

### Back navigation

When `hasUnsavedWork` is true, system back is hidden and a custom back button shows a discard confirmation. This also suppresses interactive swipe-back.

### Network pre-check

`NWPathMonitor` with 2s timeout: explicit `.unsatisfied` → offline error; timeout → proceed (URLSession handles real failures).

### Meal photo storage policy

Captured and imported photos are normalized, downsampled only when long edge exceeds 1280 px, encoded as JPEG, and persisted as a single optimized local copy in `MealEntry.photoData`. The app does not retain original full-resolution source bytes after save.

- **Initial JPEG quality:** 0.72 (`AppConstants.MealPhoto.initialJPEGQuality`), tunable after validation
- **Byte budget:** preferred ≤500 KB, soft ≤750 KB, hard ≤1 MB with quality/dimension retry
- **Floors:** do not reduce below `minJPEGQuality` or `minLongEdgePx` unless hard-cap enforcement requires it; never upscale small inputs
- **Format:** persisted photos are always JPEG regardless of input format
- **Pipeline:** `MealPhotoProcessor.prepareForAnalysisAndStorage` feeds both Gemini analysis and `photoData` on save
- **Edit path:** `loadForEditing` reuses stored bytes via `prepared(fromPersistedJPEG:)` without re-encode unless the user picks a new photo

See [`PR-04-addendum-image-storage.md`](./PR-04-addendum-image-storage.md).
