# PR9: Design System Polish & App-Wide UX

**Status:** Implemented  
**Source of truth:** [docs/technical-spec.md](../technical-spec.md) → **PR 9: Design System Polish & App-Wide UX**

## 1. Objective

Introduce asset-backed design tokens and reusable UI components, migrate in-scope feature screens to consistent chrome/colors/typography, add accessibility and motion polish, and ship app icon + launch screen assets — without changing data models, tab architecture, HealthKit behavior, or adding SPM dependencies.

## 2a. Single-user reconciliation

PR9 was finalized after the single-user simplification landed. The design-system pass is **visual and accessibility-only** — no partner flow, profile switcher, or `@AppStorage(activeUserId)` was reintroduced.

| Surface | Single-user behavior preserved |
|---------|-------------------------------|
| Dashboard | `viewModel.activeProfile`; no toolbar profile picker |
| Analytics | `profileDataRevision` refresh only; no profile menu |
| Settings | Optional display name; single delete-data flow |
| Analytics empty gate | In-screen instructional copy; no `RootView` tab hack |

## 6. Spec extensions (approved)

1. **Macro colorsets** (`Protein`, `Carbs`, `Fat`, `OnPrimary`) — asset-backed extensions beyond spec token list

4. **`CalorieProgressBand` stays in `DashboardViewModel.swift`** — `Colors.calorieProgress` mirrors thresholds; presentation-only
5. **Analytics chart first-appear animation** — deferred; weight progress chart uses opacity fade-in + one-shot count animation guard
